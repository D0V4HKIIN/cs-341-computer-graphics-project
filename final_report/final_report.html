Sea’s - 341 Captain Hook’s Black Pearl

Abstract

Our project was inspired by Sea of Thieves; a video game about pirates and their conquests in the sea. Our goal for this project was to create a realistic animation of a ship which we can navigate on the sea. The computer graphics aspect of our project has to do with the generation of waves for our infinite sea, using topics covered in the course. Furthermore, we aimed to make it as cinematic as possible by making wave-boat interaction, moving seagulls and a user interface.

Image of the game Sea of Thieves.

Technical Approach

What is Godot? Why Godot?

We used Godot, an open source game engine, because we had many ideas of features that we wanted to implement and it would not have been possible if we had to build everything from the ground up. Godot allowed us to easily work with reflections and lighting of our scene, which would not have been easily done in other frameworks, like OpenGL.

See https://godotengine.org/


Noise based Water

The waves are made using a heightmap on the vertices off a 2d mesh. The height map uses a wave function which uses as base the formula seen in the picture below. To that we add noise to the position sampling so that the waves are not just straight lines. To that we also add a small baseline noise so that it doesn’t seem to be a moving heightmap texture. Then we add up 4 waves with different speeds, heights and scales.

The documentation about shaders ([1.a], [1.b], [1.c]) has an example of how to make a water shader, from which our shader is inspired from. The values for RIM, METALLIC and ROUGHNESS that define the material have been referenced from there.

Also the fresnel and the color mixing part is replicated from a video [1.d].

The foam is made by reading the height of the “vertex” (as a varying variable) in the fragment shader and checking if it is above a certain height. If it is, we add the color sampled in the foam texture to the pixel color (which already contains the color of the water) the color of the texture of the water foam. The water foam texture is generated using noise. After some tweaking we concluded that value cubic noise using the ping pong method to combine the 5 octaves works the best.

Value cubic noise is noise generated from white noise (totally random) with cubic interpolation so make it seem less random. The cubic interpolation is the following: n = x(x(x(-a + b - c + d) + 2a - 2b + c - d) - a + c) + b, where a,b,c and d are four samples to interpolate between and x is the position in the range [0, 1] to interpolate the region between b and c( See [1.e]).

We couldn’t find any documentation explaining what the ping-pong method does so we looked into the source code of FastNoiseLite [1.f]:

The ping pong method first takes the modulo 2 of t (red lines) and then applies this pyramid shaped function to it (in green).

Infinite ocean approximation

To make the ocean seem infinite, we put together multiple plane meshes with different amounts of subdivision and scaling [2.a]. But there seems to be a bug in the engine and the dynamic subdivision doesn’t work and we had to instead have the same amount of subdivision on all planes. This way, we do not move the boat but we move the water texture in the opposite direction of the boat. Because position is a relative concept this is equivalent to moving the boat.

One problem of this technique is that it has seams between the planes. To remedy that, we scale the waves down so that the seams get so small they are not visible anymore. The downside to that is that there are no physical waves at long distances anymore, but is gets counterbalanced by the fact that it is always far away from the boat so it is not too noticeable and the normals are still computed as if the waves where not scaled down, still giving the impression to the human eye that there are waves far out.

We tried to have different amounts of subdivision using a mesh generated in blender as suggested by our TA but that had the same problems with the seams.

Also we tried generating a single circular mesh with less and less vertices further out but that ended up being a mess with lots of bugs and not looking right because the triangles were not equilaterals.

Boat

3D model

        We wanted our scene to be as creative as possible. Hence, we decided to build our own pirate ship mesh in Blender. We got ideas from a youtube video [7.b] but we also had to practice using the Blender commands and tricks. We then imported our mesh into Godot.

Flag

        In order to complete the image of our pirate ship, we added a flag on its mast. The movement of the flag is also based on noise and is implemented as a standalone spatial shader, inspired by [7.a]. We first define a displacement variable which gets the texture of the noise at a coordinate based on the UV coordinate scaled to make the noise less aggressive and add time to the x coordinate to approximate the effect of the wind. We scale this displacement by multiplying this displacement with UV.x. This way the part of the flag that touches the mast on our boat is not moving, while it moves more the further away it is from the boat. Then, for each vertex we displace it on the y-axis (the vertical one). Finally, in order to make the vertices seem like they are twisting on the x and z-axis, we displace them by the product of (displacement * distortion).

Boat Interactions

Movement

The boat uses a traditional acceleration + air/water resistance mechanism. Each time the physics are processed we add the linear/angular acceleration to the boat linear/angular speed according to the user's input. Then we multiply the linear/angular speed with the linear/angular resistance. This way the boat has some acceleration and slow down phase instead of being instantly at top speed.

Buoyancy

The boat floats on the water in a realistic manner. This is done by having 8 different points called floaters under the boat where we compute the depth of those points on the cpu (instead of the gpu for the shader). Then we apply an upwards force according to the square root of the depth. We use the square root of the depth because that way if it hits a very big wave, it will not fling up into the sky. The neat part is that the force is not applied to the center of the boat but at the floater so the boat will follow the curvature of the waves.

Water Splashes

The water splashes are a particle system included in godot that emit 2d splash images that use the billboard technique to always face the camera. We emit 10 particles when the time normalized depth difference with the last frame is above a certain limit (time normalized means that the depth is multiplied with the time delta since last frame). The initial speed of the particle is randomized but also dependent on the square root of the depth difference to make it more realistic when hitting waves with different sizes.

Noise based Clouds

Taking inspiration from the Book of shaders: Fractal Brownian Motion [6.a], we decided to implement cloud visual effects using FBM and noise.

In order to improve the performance, instead of using the Perlin noise, we decided to make use of the implementation of simplex noise by Inigo Quilez taking reference from article [6.b].

The technology we applied is the Domain Warping which applies fbm on noise multiple times to warp a fbm space within another fbm. Instead of outputting 2-dimension noise, we followed the simpler path to generate a float number. The usage of it will be explained later.

Seagulls - Boids Algorithm

The boids of birds are implemented using the classical Boids algorithm, it’s perfectly explained by article [4.b].

We’ve made use of the Godot MultiMeshInstance3D to make computation less intensive than creating individual bird mesh hundreds of times. We then store the position and velocity of all birds in two different arrays. Both position and velocities are randomly generated.

In order to limit the moving range of our birds so they don’t fly away from the scene, we added a square mesh to limit the space where they can move in. Initial position will be generated randomly but within the box, Initial velocity would be generated within the range that has been set by the users.

In our code, we implemented:

  • Separation: avoid collision between birds.

Once two boids come closer than the preset avoidance distance, we will store the separation force between them as a 3d vector.

  • Alignment: all the birds within the same group should move at the same speed.
  • Cohesion: go towards the center of the group of birds.

Both forces can be implemented in similar fashion. Once two birds come close enough to be considered as neighbors, we will store their position and velocity.

Now that we have separation, alignment and cohesion, we can calculate the steer force by combining them together. Note that we have weights for each of them so we can adjust the effect of each to achieve different visual outcomes.

However, this is not enough, we need two more things:

  1. The birds should turn around when they approach the border of the container, to achieve this, we made function calculate_boundary_force(position):

 

This makes sure once the birds enter the preset distance to the boundary, we will apply a vector to make the birds turn away from the border until they exit the range of boundaryDistance. By doing this we won’t see our birds fly away from our scene nor will they make a sharp turn or flip when they hit the container boundary.

  1. The groups that have been formed should be able to break again, otherwise we will end up with one or multiple huge groups which the birds can enter but never quit.

To do so, we added randomly spawned invisible attractors within the container, and assign them stronger force but small detection distance so that they can attract the portion of the groups that are close enough to it, making them exit from the original group.

Now that we have everything, we simply add them together with the original velocity of each bird and update the velocity array with new velocity, and calculate and update the position array with the new velocity we just obtained.

The last few lines of our boids implementation simply ensure the birds will look at the front of them and make rotation accordingly instead of plain translation without the birds looking at the direction they move to.

Cameras

        In order to make the final video for our project, we decided that we would like to have several camera shots of our scene, following different paths and focusing on different aspects of our ocean.

        First, we implemented a “Rotating camera”, which simply rotates around its target in order to catch all its angles. We assign it a rotational speed, a radius and a target. Then, for each frame we alter its x and z coordinates so that it performs this rotational movement.

        Next, we made a Bezier curve camera.This camera has the boat as the target and looks at it all the time while moving. The path of the camera is determined by 4 points in our 3D space, two of

which are the start and end points. The camera uses a function called “cubic_bezier” which interpolates through the points using the cubic Bezier curve formula. We implemented the interpolation and hence the cubic_beizer function based on the lecture. Moreover, in the main function (called _process), we perform two bezier curve paths: one with points p0, p1, p2, p3 and one which starts where we ended and follows the curve of p3, p5, p4, p0. This way, the camera movement finishes at the starting point and we can loop these movements in a continuous curve.

        Finally, we implemented a double Bezier camera. The code for the movement of this camera is identical to the one above, however the target is a moving_target. More specifically, we defined the movement of an invisible target which follows a different Bezier curve (again using the “cubic_bezier” function and using interpolation) and assigned the camera to, while moving, keep looking at the invisible target. This way, we got a very cinematic shot of our scene, almost as if the camera was on one of the seagulls.

Results

        Our final result is close to our initial goal. We managed to produce realistic random waves which reflect the sky and clouds and which all move to one direction, simulating realistic wind-waves movement. Our waves have foam when they are tall enough and the foam follows a very convincing movement, having a “changing shape” as the waves move.

        We were very happy to have managed a type of infinite ocean, which means unlimited game/interaction time, as the boat can travel on the sea forever. Moreover, the boat interacts with our ocean and creates splashes that make our scene more realistic. The boat we built ourselves in Blender and hoped to make it similar to a pirate ship.

Furthermore, we made our scene interactive by placing a boat which can be moved using the JKIL keys of the keyboard and by placing a moving camera which is controlled by the WASD keys. Also, the player can use sliders to alter the height waves, the wave speed and the bird count. For a greater experience, we also allow the user to hide the UI screen and easily toggle fullscreen.

        Finally, we managed to create a realistic cloudy sky and we can easily change sky and sea colors from the Godot settings. Our last extra feature is the seagulls which we made them flap their wings and follow the boids algorithm in the sky. In our scene, it takes some time for the birds to join a swarm.

Problems encountered

Our initial core project contained the generation of waves using Fast Fourier Transforms, following closely the paper written by J. Tessendorf titled “Simulating Ocean Water”. However, we faced several problems while approaching the Fourier Transform waves. Firstly, we lost a lot of time getting familiar with the concept of Fourier Transforms, as it was new to all of us. We were struggling with the idea of how to represent the height of waves using real and imaginary numbers and how to implement that in the shader.

        In the end, we consulted with our TA and decided that since the result of the wave generation would not make that much of a difference, we could focus on other aspects of our implementation and keep the waves as noise generated.

Further research

        Regarding the wave generation, we all agree that, given more time, it would be possible to work on Tessendorf waves and make our sea more realistic. It would also be possible to make the waves more choppy or add wind contribution to our scene.

        Additionally, we would have liked to work on the “horizon” of our scene. In our current implementation, we notice the corners of our square ocean 3d node and we would have liked to either add some curvature or maybe make a circular mesh so that the horizon is similar to one in the ocean.

        Another improvement would be to add weather conditions to the scenes, such as rain, thunder and lightning. We would like to be able to generate several ocean conditions for our pirate ship, like calm seas, high waves due to wind, storm situation or night-time.

        Finally, we could have procedurally generated islands and simulate the waves on the shore.
